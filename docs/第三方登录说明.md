# 第三方登录 - Google 后端处理说明

## 前端传递的数据

前端 Google 登录成功后，会调用统一的社交登录接口：

```
POST /auth/social/login
```

请求体：

```json
{
  "socialType": 3,           // 3 = Google
  "code": "xxx",             // Google 授权码（一次性，每次不同）
}
```

## 后端处理流程

### 第一步：用 code 换 access_token

Google 的授权码 `code` 不能直接用于获取用户信息，需要先换成 `access_token`。

```
POST https://oauth2.googleapis.com/token
Content-Type: application/x-www-form-urlencoded

code={前端传来的code}
&client_id={你的Google Client ID}
&client_secret={你的Google Client Secret}
&redirect_uri={必须与前端一致}
&grant_type=authorization_code
```

响应：

```json
{
  "access_token": "ya29.xxx",
  "expires_in": 3600,
  "refresh_token": "1//xxx",
  "scope": "openid profile email",
  "token_type": "Bearer"
}
```

### 第二步：用 access_token 获取用户信息

```
GET https://www.googleapis.com/oauth2/v2/userinfo
Authorization: Bearer {access_token}
```

响应：

```json
{
  "id": "1085xxx",           // 用户的唯一标识，重要！
  "email": "user@gmail.com",
  "name": "User Name",
  "picture": "https://lh3.googleusercontent.com/xxx"
}
```

### 第三步：关联/创建用户

1. **查询**：用 `email` 或 `id`（推荐用 `id`，更稳定）在用户表中查找是否已存在
2. **不存在**：创建新用户，记录 `google_id = "1085xxx"`, `email`, `avatar` 等
3. **存在**：更新用户的最后登录时间、头像等信息
4. **返回**：生成自己的 JWT Token 返回给前端

## 关键点

| 字段 | 说明 |
|------|------|
| `code` | 一次性，每次授权都不同，**不能用来唯一标识用户** |
| `access_token` | 短期有效，用来换取用户信息 |
| 用户唯一标识 | 应使用 Google 返回的 `id` 字段，不是 email（用户可能换邮箱） |

## 为什么不能用 code 作为用户标识？

1. **一次性**：每次点"用 Google 登录"都会生成新的 code
2. **短期有效**：几分钟就会过期
3. **不可逆**：code 只能换一次 token，换完就失效

所以必须用 `code` → 换 `access_token` → 调用户信息 API → 获取唯一的 `id` → 用这个 `id` 来关联用户。

## 统一登录接口的 socialType 对照

| socialType | 平台 |
|------------|------|
| 1 | 微信 |
| 2 | QQ |
| 3 | Google |
| 4 | Apple |
| 5 | 抖音 |

后端根据 `socialType` 字段选择不同的处理逻辑即可。
